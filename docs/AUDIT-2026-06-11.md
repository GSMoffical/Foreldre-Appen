## TL;DR

The app is in solid shape and the previous audit's **critical** finding (`deleteAllEventsForUser`) is **fixed**. Mutations now correctly route through `SECURITY DEFINER` RPCs, invite/token handling is strong, Sentry is GDPR-careful, account deletion is complete, and the test suite (560 passing) + clean typecheck/build are real signs of health.

The most important remaining issues are not in the React code — they're in **deployment and database policy**:

1. **Security headers (CSP, X-Frame-Options, …) are not actually served in production.** They live in a `_headers` file that Hostinger ignores; the real `.htaccess` only sets charset.
2. **A linked co-parent can still DELETE any event/task they don't own** via a direct API call — RLS is still `FOR ALL`.
3. **No global error boundary** — a single render throw white-screens the whole app.
4. **Two internal files (`vibe_coding_wip.docx`, `morten pack.zip`) sit in `public/`** and are published to the live site.

## Findings by severity

### HIGH

**H1 — Security headers are not served in production**
`public/_headers` contains a strong CSP plus `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`. That file format is **Netlify/Cloudflare Pages only**. The site deploys to **Hostinger (Apache/LiteSpeed)**, where `public/.htaccess` is the only thing honored — and it contains just `AddDefaultCharset UTF-8`. Result: **no CSP, no clickjacking protection, no nosniff in production.**
*Fix:* Port the headers into `.htaccess` via `<IfModule mod_headers.c>` `Header set …` directives (and `Header always set` for CSP). Then re-verify with `curl -I`. Tighten `script-src` while you're there — drop `'unsafe-inline'` for scripts (Vite doesn't need it).

**H2 — Linked co-parent can DELETE/overwrite events & tasks they don't own**
`supabase-invites.sql:147` (events) and `supabase-tasks.sql:52` (tasks) still use `FOR ALL` with an "owner OR linked" predicate. The `events_keep_owner` trigger only prevents *ownership reassignment on UPDATE* — it does **not** stop DELETE. So an invited partner can open the console and run `supabase.from('events').delete().eq('user_id', '<owner-id>')`-style calls and wipe rows they never created. Data loss, no audit trail. (This is the prior audit's Issue #2/#3, only partially mitigated.)
*Fix:* Split `FOR ALL` into explicit policies — partners get `SELECT` (+ `INSERT` if desired), but `DELETE`/`UPDATE` restricted to `user_id = auth.uid()`. The app already mutates through owner-checked RPCs, so this should be low-friction.

**H3 — No global React error boundary**
`main.tsx` wraps `<App/>` in providers but no error boundary. Only `TankestromImportPreviewBoundary` exists, scoped to one feature. Any render exception anywhere else → blank white screen, and Sentry can't capture React render errors well without a boundary.
*Fix:* Wrap `<App/>` in `Sentry.ErrorBoundary` (or a small custom one) with a friendly Norwegian fallback + "last inn på nytt" action.

### MEDIUM

**M1 — Internal files published to the live site**
`public/morten pack.zip` and `public/vibe_coding_wip.docx` are copied into `dist/` by Vite and uploaded by the SFTP deploy path — i.e. fetchable at `https://<site>/vibe_coding_wip.docx`. Remove them from `public/` (and from the host if already deployed).

**M2 — Redundant polling on top of realtime (cost + battery)**
Every active client runs `setInterval` refetches: events **every 10s** (`useScheduleState`), family **every 10s** (`FamilyContext`), tasks every 30s — *and* also holds Supabase realtime subscriptions for the same tables. The polling is largely redundant with realtime and multiplies DB reads/egress per user and drains mobile battery.
*Fix:* Treat realtime as primary; keep polling only as a coarse fallback (e.g. 60s) or trigger refetch on `visibilitychange`/reconnect only. The visibility-based refresh is already there — lean on it.

**M3 — Deploy mode silently changes what ships**
`deploy-hostinger.ps1`: the **SFTP/key** path uploads all of `dist/` (so `.htaccess`, `_headers`, junk files all go up), but the **`-UsePasswordAuth`/scp** path uploads only `index.html`, `favicon.svg`, `manifest.webmanifest` + `assets/`. That means `.htaccess` (and any future headers config) is **not** uploaded in password mode — config diverges by how you deployed.
*Fix:* Make both paths upload the same explicit allowlist of root files (including `.htaccess`), and exclude the junk files explicitly.

**M4 — `events.person_id` has no referential integrity**
`supabase-setup.sql:9` — `person_id text` with no FK/constraint to `family_members`. Events can reference non-existent or cross-family member IDs (prior Issue #4, still open). Add a validating trigger/CHECK in the insert/update RPCs that the `person_id` belongs to the owner's `family_members`.

**M5 — No optimistic-locking / concurrent-edit detection**
Two devices editing the same event/task → last write silently wins (prior Issue #8, still open). `tasks` has `updated_at`; add `.eq('updated_at', known)` guards (and an `updated_at` on events) to detect conflicts and refetch instead of clobbering.

### LOW / quality

- **L1 — `App.tsx` is a 926-line god component** (~30 `useState`/`useRef`, many effects, all the toast/onboarding/tankestrom orchestration). Hard to test and reason about; extract toast manager, onboarding gating, and tankestrom-import wiring into hooks/components.
- **L2 — Two mega-files:** `useTankestromImport.ts` (**6,221 LOC**) and `TankestromImportDialog.tsx` (**5,720 LOC**). This is the maintainability/risk hotspot of the codebase (and where most of the 560 tests are aimed — for good reason). Plan incremental decomposition.
- **L3 — Dead code:** `fetchEventsForDateRange` and `fetchEventsForDate` in `eventsApi.ts` are unused (app uses the `get_events_for_calendar` RPC). Remove to avoid confusion (they're also the unbounded-query path the old audit worried about).
- **L4 — `get_events_for_calendar` has no `LIMIT`** within its date range. Range-scoping makes this low risk for normal windows; add a safety cap (e.g. `LIMIT 2000`) anyway.
- **L5 — Three overlapping onboarding systems:** `OnboardingFlow`, `OnboardingTour`, and the new (uncommitted) `GettingStartedChecklist`. Recent commits show remove-then-readd churn. Confirm which are intended and delete the rest.
- **L6 — ~200 ungated `console.*` calls** (mostly `console.error` in catch blocks) run in production. Route through a tiny logger that no-ops outside DEV (except genuinely actionable errors → Sentry).
- **L7 — `date-holidays` is 1.35 MB (221 KB gzipped)** — a world-wide holiday DB for a Norway-only app. It's lazy-loaded (good) but the calendar needs it almost immediately. Replacing it with a ~30-line Norwegian holiday calculator (fixed dates + Easter-relative) removes ~1.3 MB from the wire.
- **L8 — Nested git repo `Foreldre-Appen/`** (own `.git`, unrelated "Initial commit") sits inside the project, and `.gitignore` has a UTF-16-mangled `Foreldre-Appen/` line. Remove the stray nested repo; fix the ignore entry encoding.
- **L9 — RPCs granted to `anon`** (`get_events_for_calendar`, `update_event_keep_owner`, …). They internally null-check `auth.uid()`, so it's mitigated, but there's no reason to expose them to `anon` — grant to `authenticated` only.
- **L10 — Dev-only deps:** 2 moderate advisories (esbuild/vite dev server). Not production-affecting; bump Vite when convenient.
- **L11 — A11y:** one `<img>` without `alt` in `OnboardingFlow.tsx:211`. Add alt text (or `alt=""` if decorative).
- **L12 — `.env.example`** has stray shell `echo … >> .env.example` lines appended into the file body (copy/paste artifact). Clean up.
